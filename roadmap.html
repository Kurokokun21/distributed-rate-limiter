Build Roadmap

Distributed Rate Limiter

Six phases, each one a working system. Docker skeleton → Redis cache → atomic token-bucket limiter → Celery job queue → WebSocket push → Locust load test. Build strictly in order — every phase depends on the plumbing laid down by the one before it.

3–4 focused weeks, part-time 6 phases 1 non-optional measurement phase
P0 P1 P2 P3 P4 P5

bar width = rough time budget · colour = the one new piece of tech each phase introduces

How to use this

  • Build strictly in order. Can't cache before the skeleton runs, can't push job updates before jobs exist.
  • Every phase ends working. You're never stuck half-broken — each box is a deployable checkpoint, not a fragment.
  • Learn one technology at a time. Don't try to hold the whole stack in your head before you've touched Redis.
PHASE 0

Skeleton & Plumbing

docker compose

Three containers learning to talk: an API service, Redis, and a placeholder worker, wired together in one docker-compose.yml. A single health-check endpoint proves all three can reach each other before any real logic exists.

~0.5 day docker compose up boots all three; /health returns 200 from each
PHASE 1

Cache Layer

redis

Wrap a slow endpoint in a cache-aside pattern — read-through, TTL expiry, invalidation on write. A rate limiter is a cache with opinions, so Redis's data structures and eviction behaviour need to be second nature before limiter logic goes on top.

~1 weekend cached endpoint measurably faster on repeat hits; TTL expiry verified under test
PHASE 2

The Rate Limiter

token bucket · lua

Token-bucket rate limiting as an atomic Redis Lua script — no race condition lets two concurrent requests both slip through on the last token. This is the core algorithm the whole project is named after.

~1 weekend concurrent load never over-admits; script proven atomic under a race test
SLOW DOWNthis is the interview-gold concept — understanding atomic distributed state, not just calling a library. Don't rush it.
PHASE 3

Background Jobs

celery

Move rate-limited work off the request path into Celery workers, using Redis as the broker. Real systems don't block a client on slow work — they queue it and hand back a ticket.

~1 weekend submitting a job returns immediately; job finishes async; status is queryable
PHASE 4

Live Updates

websockets

Push job-status changes to the client over a WebSocket instead of polling. This closes the loop end to end: request in, queued, processed, result pushed back — the full real-time pipeline.

~a few days client sees job status update live, with zero polling
PHASE 5

Prove It

locust

Load-test the whole system with Locust: throughput, latency percentiles, and limiter behaviour under real concurrency. "It works" isn't a resume line — "held 500 req/s at p99 under 40ms" is.

~a few days a latency & throughput report with real numbers
NOT OPTIONALthis is where the resume-worthy proof comes from. Stopping at "it works" skips the entire point.
